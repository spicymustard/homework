# RAG 구축 계획 (file1.pdf)

## 목표
KT 인터넷 이용약관(file1.pdf)을 기반으로, 고객 질문에 답변하는 RAG 시스템 구축

---

## 사용 기술

| 역할 | 도구 |
|---|---|
| PDF 읽기 | PyMuPDF (fitz) |
| 텍스트 분할 | RecursiveCharacterTextSplitter |
| 임베딩 | OpenAI text-embedding-3-small |
| 벡터 DB | FAISS (로컬 저장) |
| LLM | GPT-4o-mini |
| 프레임워크 | LangChain (LCEL 방식) |
| 인터페이스 | Gradio (웹 챗봇 UI) |

---

## 단계별 계획

### 1단계 — PDF 로드
- `data/pdf/file1.pdf` 파일을 PyMuPDF로 읽어 전체 텍스트 추출
- 예상 결과: 약 185,000자

### 2단계 — 텍스트 분할 (Chunking)
- 추출한 텍스트를 일정 크기로 잘라 검색 가능한 단위로 분리
- `chunk_size=500`, `chunk_overlap=50` 사용
- 예상 결과: 약 500개 청크

### 3단계 — 임베딩 및 벡터 DB 생성
- 각 청크를 OpenAI 임베딩 모델로 숫자 벡터로 변환
- FAISS 벡터 DB에 저장 → `faiss_index/` 폴더에 보관
- 이후 실행 시 재생성 없이 로드하여 사용

### 4단계 — 멀티턴 RAG 체인 구성
- **질문 재작성 (Contextualize)**: 이전 대화 이력 + 새 질문을 보고, 문맥 없이도 이해되는 독립적인 질문으로 재작성
  - 예) "그럼 그게 얼마예요?" → "인터넷 해지 시 할인반환금은 얼마인가요?"
- **검색 (Retrieval)**: 재작성된 질문으로 벡터 DB에서 관련 청크 4개 검색
- **답변 생성 (Generation)**: 검색된 청크 + 대화 이력 + 질문을 프롬프트에 담아 GPT-4o-mini에 전달
- 약관에 없는 내용은 "확인되지 않습니다"로 답변하도록 프롬프트 설정
- 대화 이력은 세션 내에서 유지 (새로고침 시 초기화)

```
이전 대화 이력 + 새 질문
        ↓
   질문 재작성 (LLM)
        ↓
   벡터 DB 검색
        ↓
 대화 이력 + 검색 결과 + 질문 → GPT → 답변
        ↓
   대화 이력에 추가
```

### 5단계 — Gradio 웹 인터페이스
- 브라우저에서 바로 사용할 수 있는 챗봇 UI 제공
- 대화 이력이 화면에 누적되는 채팅 형식
- `python3 rag.py` 실행 후 localhost URL로 접속

### 6단계 — 실행 및 테스트
- 멀티턴 테스트 예시:
  - "인터넷 서비스 해지 시 위약금이 있나요?"
  - "그럼 그게 얼마예요?" ← 이전 답변 문맥을 이어받아야 함
  - "일시정지는 어떻게 신청해요?" ← 새로운 주제로 전환

## 검토 요청 사항
- 청크 크기(500자)가 적절한지 여부
- 검색 개수(k=4)를 늘릴지 여부
- 테스트 질문 추가 또는 변경 원하시면 말씀해 주세요
